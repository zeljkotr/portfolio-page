DEVOPS · CONTAINERIZATION & CI/CD

Dockerized Monitoring Tools with Automated CI/CD Pipelines

TYPEPersonal / portfolio projects
ROLESole developer, end-to-end
STACKPython, Docker, GitHub Actions, Docker Hub
REPOSgithub.com/zeljkotr

Challenge

Writing a working Python script is one thing — shipping it the way production tooling actually gets shipped is another. The goal was to move two monitoring scripts from "runs on my machine" to a repeatable pipeline: containerized, version-controlled, automatically built and tested, and automatically published — the same delivery model used in real DevOps roles, not a one-off deploy.

Project A — Server Ping Monitor with Email Alerts

A Python monitor (subprocess, datetime, smtplib) that pings a list of servers on a schedule and emails an alert the moment one goes unreachable. Set-based deduplication tracks which hosts are already known-down, so a flapping link triggers one alert instead of flooding the inbox every cycle. Windows-specific fixes — TTL-based reachability checks, CREATE_NO_WINDOW to keep it silent as a background task, and explicit UTF-8 encoding — keep it reliable across both the Windows and Linux hosts it runs on.

Project B — Flask System-Monitor Dashboard

A small Flask app (app.py, monitor.py, alerts.py, config.py) serving a live web dashboard of CPU, RAM, disk, network throughput, uptime, temperature, the top 5 processes by resource use, and active SSH sessions — running continuously on the Ubuntu server and reachable from any device on the network.

Key Decisions

Docker over bare scripts: both tools ship as images, not loose Python files — the same environment on the Windows dev machine, the Ubuntu server, and anywhere else the image gets pulled.

GitHub Actions over manual builds: every push to main triggers a workflow that builds the image and pushes it straight to Docker Hub — no manual docker build / docker push step to forget or get wrong.

Alert deduplication over raw ping-and-email: a naive version of the monitor would re-alert every cycle a host stays down; tracking already-alerted hosts in a set turned a spammy script into one that respects the inbox.

Outcome

Two working, Dockerized tools with fully automated pipelines: every commit triggers a GitHub Actions build that pushes a fresh image to Docker Hub, ready to pull and run anywhere. Both are part of a structured, ongoing transition from systems engineering into DevOps — applying container and pipeline practices to real, running infrastructure rather than tutorial exercises.

VIRTUALIZATION · PRODUCTION MIGRATION

Windows 10 Physical-to-Virtual Migration onto TrueNAS / KVM

ORGMTC Nissal d.o.o.
TIMEFRAMEApr 2026
ROLESole engineer, end-to-end
STACKKVM, QEMU, TrueNAS SCALE, ZFS, VirtIO

Challenge

A production Windows 10 machine needed to move off ageing physical hardware and onto a virtualized, ZFS-backed platform — without downtime-causing data loss, and without losing driver or boot compatibility along the way.

Approach

  • Captured a consistent disk image of the live system using Disk2vhd with VSS, avoiding the need to take the source machine offline during imaging.
  • Converted the resulting VHDX image to RAW format via QEMU, the format best suited to direct ZFS-backed storage on TrueNAS SCALE.
  • Provisioned and tuned a ZVOL on ZFS for the new virtual disk, balancing size, performance, and snapshot overhead.
  • Imported the converted image into the ZVOL using low-level dd operations for a byte-accurate transfer.

Key Decisions

RAW over VHDX: RAW avoids the translation overhead of a container format and integrates directly with ZFS block storage, which mattered more here than VHDX's portability features.

VirtIO over emulated hardware: after the migrated VM failed to boot (UEFI/BIOS mismatch, missing bootloader), I diagnosed a driver mismatch between the physical machine's hardware and the VM's emulated devices, and resolved it by installing VirtIO drivers — trading a small amount of extra setup for significantly better disk and network performance going forward.

Outcome

Delivered a fully functional virtual machine with networking and services restored, gaining portability, easier backup/snapshot management, and a path to better resource scaling than the original physical host allowed.

AI INFRASTRUCTURE · SELF-HOSTED INFERENCE

Self-Hosted LLM Inference Platform (Llama + Open WebUI on TrueNAS)

ORGMTC Nissal d.o.o.
HOSTTrueNAS SCALE (on-prem, live production network)
ROLESole engineer, end-to-end
COMPUTECPU-only inference
STACKOllama, Open WebUI, Docker Compose

Challenge

Run a local large language model entirely on existing TrueNAS hardware — no GPU, no cloud API calls — while keeping the deployment simple enough to maintain, update, and access from any device on the network through a clean web interface.

Approach

  • Deployed Ollama as the model runtime, handling model downloads, quantization formats, and the local inference API.
  • Deployed Open WebUI as the front end, connected to Ollama's API, for a ChatGPT-style interface accessible from any browser on the network.
  • Defined both services in a single Docker Compose stack on TrueNAS SCALE, with persistent storage backed by a dedicated ZFS dataset.
  • Selected quantized model sizes that ran at usable response speed on CPU-only hardware.

Key Decisions

CPU-only inference: rather than holding off until GPU hardware was available, I deployed on existing CPU resources and tuned model size/quantization to keep response times practical — trading peak performance for a working system today.

Docker Compose over TrueNAS Apps: chose a hand-written Compose stack for direct control over volumes, networking, and upgrade timing, rather than the more opinionated built-in app catalog.

Extended Capability — IoT Network-Monitoring Agent

Beyond a basic chat interface, I extended the assistant into a voice-driven agent deployed in the company's IoT sector, wired directly to network checks across the live company infrastructure — asking it out loud to check disk usage across all computers on the LAN, or to flag a user whose RAM or CPU usage is running high. It performs automatic, on-request surveillance of the complete internal network, moving the deployment from a Q&A chatbot toward a genuine agentic monitoring tool running in production, not a lab demo.

Outcome

A self-hosted, privacy-respecting LLM chat platform running entirely on company-owned hardware, with no per-token cost or external API dependency, extended into a voice-driven IoT agent that automatically monitors the complete internal network in production.

INDUSTRIAL SYSTEMS · LEGACY MODERNIZATION

Legacy SCADA System Virtualization (Windows XP → VMware)

ORGMTC Nissal d.o.o.
SYSTEMPressezi SCADA
SOURCEObsolete Windows XP host
ROLESole engineer, end-to-end
STACKVMware vCenter Converter, VMware Workstation

Challenge

An industrial SCADA system was still running production on a physical Windows XP machine — an OS long past end-of-life, on hardware that could fail at any time with no replacement available. The SCADA software couldn't be upgraded or replaced without risking compatibility with the PLC equipment it controlled.

The Real Decision

The core engineering call wasn't a technical migration step — it was choosing to virtualize and preserve the exact original software environment rather than attempt to replace or upgrade the SCADA software. Any change risked breaking PLC communication built around very specific, undocumented timing and driver behavior. Virtualization let me eliminate the hardware risk entirely while leaving the proven software untouched.

Approach

  • Performed a P2V conversion using VMware vCenter Converter, finalized in VMware Workstation.
  • Configured virtual hardware, networking, and communication interfaces to remain fully compatible with existing industrial equipment.
  • Set up serial/USB device passthrough so the VM kept communicating with physical PLC hardware exactly as the original machine did.
  • Migrated network settings and IPs, validated full SCADA functionality before cutover, and implemented a backup/snapshot strategy the physical setup never had.

Outcome

Eliminated hardware dependency and single-point-of-failure risk, added proper backup/disaster recovery for the first time, and extended the system's operational lifecycle — all without touching the original, proven software environment.

Key Lessons

  • Virtualization as risk mitigation, not modernization: the goal was never to modernize the SCADA software — it was to eliminate the one thing that could bring everything down. Keeping the software frozen was the right call.
  • Serial/USB passthrough is the critical path: the VM can run all day, but if PLC communication breaks, nothing works. Testing this first, before cutover, was essential.
  • Snapshots are not backups, but better than nothing: the original setup had zero recovery options. Even a basic snapshot policy meant recovery in minutes instead of a physical rebuild.
NETWORK ENGINEERING · GREENFIELD DESIGN

Enterprise Network Deployment on MikroTik

ORGMTC Nissal d.o.o.
TIMEFRAMEMar 2022
SCALE100+ devices/users, 6 sites
STACKMikroTik CCR2116, CRS326, RB260G

Challenge

Design and deploy a network from scratch: a main hub housing servers and internet uplink, with six separate locations needing reliable, segmented connectivity back to that hub — plus remote access for staff and headroom to monitor and troubleshoot without on-site visits.

Approach & Decisions

Star topology with central fiber distribution, chosen for simpler fault isolation than a daisy-chained layout — a single link failure stays contained to one branch instead of cascading through the chain.

All-fiber links end-to-end, including inter-building runs, to keep the topology consistent, avoid copper distance limitations, and simplify future troubleshooting.

VLAN segmentation to separate traffic classes per site/department and limit the blast radius of any single compromised device.

Layered remote access: WireGuard for fast, modern site-to-site/staff access, with OpenVPN and L2TP kept available for client compatibility.

Configuration Highlights

  • Firewall rules: separated into input, forward, and output chains per interface group; default-deny posture with explicit allow rules per VLAN.
  • VPN segmentation: WireGuard peers issued individual keys with scoped routing — staff remote access reaches only the subnets they need.
  • QoS: Simple Queue and PCQ fair-queue bandwidth per user at peak hours, preventing any single workstation from saturating the inter-site fiber link during production shifts.
  • Configuration backup: automated MikroTik export scheduled on all devices, written to a central share — the network is recoverable from scratch without on-site access.

A documented, segmented, remotely-manageable network spanning a main hub and six production sites with 100+ devices — replacing entirely ad-hoc, undocumented connectivity.

RF ENGINEERING · INDEPENDENT R&D

Meshtastic LoRa Mesh Network — Device Engineering & RF Optimization

TIMEFRAMEFeb 2025 – Present
HARDWAREESP32, LoRa (868 MHz ISM)
TOOLSMeshtastic firmware, Jugomesh

Challenge

Real-world RF behavior rarely matches datasheet promises. The goal was to understand the actual limits of long-range, low-power LoRa mesh communication — not in theory, but by building, deploying, and measuring real nodes in real conditions.

Approach

  • Engineered and configured Meshtastic nodes on ESP32 hardware with 868 MHz LoRa transceivers.
  • Designed and iterated on antenna setups, testing how placement and orientation changed propagation and range.
  • Field-tested message routing, latency, and reliability across rural and urban scenarios, in line-of-sight and non-line-of-sight conditions, feeding measured data (via Jugomesh) back into node placement decisions.

Outcome

An iteratively-tuned mesh network with measurably improved coverage and resilience, grounded in field data rather than assumptions — plus a practical understanding of where RF theory and real-world deployment diverge.

DATA TOOLING · PYTHON & WEB

Meshcore Data Collection & map.msh.rs

RELATED TOMeshtastic LoRa mesh, above
ROLESole engineer, end-to-end
STACKPython, Cloudflare Tunnel, map.msh.rs

Challenge

Field-testing the mesh network produced telemetry from USB companion devices, but no easy way to see network health at a glance — reading raw logs node-by-node doesn't scale, and the network needed to be visible and shareable without exposing anything insecurely to the internet.

Approach

  • Wrote Python scripts to collect and process telemetry from USB companion devices in the field.
  • Built map.msh.rs, a live web map visualizing repeater and companion node status across the mesh.
  • Exposed the map securely via a Cloudflare tunnel, avoiding open inbound ports or a public-facing server.

Outcome

Real-time visibility into mesh network health from any browser, replacing manual log-checking with a live, shareable map — accessible over a secure tunnel with no exposed infrastructure.
