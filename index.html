Bailiff Sentinel — Financial Anomaly Detection, Run Like Production

Environments
Dev/test + isolated production host
Repo
github.com/zeljkotr/bailiff-sentinel

Challenge

Bank statement exports (XML, inconsistent schema and encoding depending on export settings) needed to be checked for a specific fraud signature: money paid out to one account, then quietly returned by the bank days later — usually a sign of a misdirected transfer. Any tool touching real account data had to run somewhere that could never be exposed to the open internet, while still being reachable for daily use and safe to update without risking the data it holds.

What Makes This an Ops Project, Not Just a Script

  • Schema-agnostic parsing: auto-detects and flattens arbitrary/nested XML structures instead of hardcoding field names, and self-corrects encoding (UTF-8, UTF-16, CP1250) since bank exports arrive inconsistently encoded from different branches.
  • Dev/prod separation: a development instance for testing changes, fully isolated from the production host handling real financial data — standard practice, applied here deliberately rather than skipped for a "personal" project.
  • systemd-managed service: runs as a proper unit (Restart=on-failure, enabled at boot), not a terminal session someone has to remember to restart — survives reboots and crashes unattended.
  • Zero public exposure: the production host has no open inbound ports. Access is entirely over a Tailscale (WireGuard) mesh — reachable from any authorized device, with nothing listening on the public internet to attack.
  • Gated CI/CD: GitHub Actions builds and validates on every push, but deployment to the box holding real data requires a manually triggered workflow — a push to main is never automatically "go live."
  • Idempotent schema migrations: the database checks and adds any missing columns on startup, so upgrades apply safely across versions with no manual migration step and no downtime.
  • Fails loud, not messy: every error path returns structured JSON, never a raw stack trace or an HTML error page — and debug mode is explicitly off, since a debug console left reachable is a standing risk even on a private network.

Deployment Pipeline

git push origin main → GitHub Actions: build + syntax check [ OK ] (deploy job waits for manual approval — production is gated) → operator clicks "Run workflow" → Tailscale OAuth: ephemeral, tag-scoped node joins tailnet [ OK ] → SSH over private mesh IP → git pull, pip install, systemctl restart [ OK ] → ephemeral CI node disconnects, zero standing access left behind

Key Decisions

systemd over a screen session
Crash recovery and boot-persistence without a human remembering it exists.
Tailscale over a traditional VPN or open port
Reachable from every authorized device — and from CI — without a static IP, a VPN client, or a single inbound port exposed to the internet.
Manual deploy gate over full automation
Production holds real account data. A push to main should never be the same action as "go live."
Dev/prod split over one shared instance
New checks and UI changes get validated on a throwaway instance before they ever touch the data that matters.

Detection Capabilities

Beyond the operational shape, the tool itself runs five configurable checks — full and partial refund pairing (matched by reference number and amount), duplicate transaction IDs, duplicate payments, and statistical outlier detection — with persistent upload history, manual false-positive marking that's excluded from future reports, and CSV/PDF export (custom-rendered reports with embedded font support).

Outcome

A financial-anomaly detector that runs unattended in production: restart-proof, reachable only over a private mesh network, deployable on demand through a gated pipeline that leaves no standing credentials or open ports behind. The same operational shape as a small production microservice — not a personal script that happens to be running somewhere.

Challenge

Writing a working Python script is one thing — shipping it the way production tooling actually gets shipped is another. The goal was to move two monitoring scripts from "runs on my machine" to a repeatable pipeline: containerized, version-controlled, automatically built and tested, and automatically published.

Project A — Server Ping Monitor with Email Alerts

A Python monitor (subprocess, datetime, smtplib) that pings a list of servers on a schedule and emails an alert the moment one goes unreachable. Set-based deduplication tracks which hosts are already known-down, so a flapping link triggers one alert instead of flooding the inbox every cycle.

Project B — Flask System-Monitor Dashboard

A small Flask app serving a live web dashboard of CPU, RAM, disk, network throughput, uptime, temperature, top-5 processes, and active SSH sessions — running continuously on the Ubuntu server and reachable from any device on the network.

git push → main → GitHub Actions: build image [ OK ] → push to Docker Hub [ OK ] → pull & redeploy container, anywhere
Docker over bare scripts
Same environment on the dev machine, the Ubuntu server, and anywhere else the image gets pulled.
GitHub Actions over manual builds
Every push to main builds and publishes — no manual docker build/push to forget or get wrong.

Outcome

Two working, Dockerized tools with fully automated pipelines: every commit triggers a build that publishes a fresh image, ready to pull and run anywhere.
