DevOps · Production Deployment · Zero-Trust Access

Bailiff Sentinel — Financial Anomaly Detection

Role Sole engineer — design, build, deploy, operate Stack Python, Flask, SQLite, systemd, Tailscale, GitHub Actions Repository github.com/zeljkotr/bailiff-sentinel

A web application that analyzes Serbian bank statement exports (XML, inconsistent schema and encoding across branches) to detect a specific anomaly: a payment sent to one account, then quietly returned by the bank days later — usually a sign of a misdirected transfer. Because the tool handles real account data, it was built and operated with the same discipline as a small production service, not a personal script.

Design and operational decisions

  • Schema-agnostic parsing — automatically detects and flattens nested XML structures rather than hardcoding field names, and self-corrects encoding (UTF-8, UTF-16, CP1250) since exports vary by branch and settings.
  • Separate development and production environments — changes are validated on a test instance before reaching the server handling real financial data.
  • Managed as a systemd service — runs continuously, restarts automatically on failure, and survives server reboots without manual intervention.
  • No public network exposure — the production host has no open inbound ports; access is entirely through a Tailscale (WireGuard) mesh network.
  • Gated deployment pipeline — GitHub Actions validates every change, but deployment to production requires a manually triggered workflow rather than deploying automatically on every push.
  • Automatic, idempotent schema migrations — the database adds any missing columns on startup, so upgrades apply safely with no manual migration step or downtime.
  • Structured error handling — every error path returns a clean response rather than a stack trace, with debug mode explicitly disabled.

Deployment pipeline

  1. Change is pushed to the main branch; GitHub Actions runs a Python syntax check automatically.
  2. Deployment to production is a separate, manually triggered step.
  3. The pipeline authenticates to the private network using short-lived, tag-scoped OAuth credentials — no standing VPN keys or open ports.
  4. It connects over SSH via the private network, pulls the latest code, installs dependencies, and restarts the service.
  5. Temporary credentials are discarded immediately after the run.

Key decisions

  • systemd over a manually run processEnsures the service recovers from crashes and survives reboots without anyone needing to remember to restart it.
  • Tailscale over a traditional VPN or open portReachable from every authorized device, including CI, without a static IP or any port exposed to the public internet.
  • Manual deployment gateProduction holds real financial data — a code push should never be equivalent to "go live."

The application itself runs five configurable checks (full and partial refund matching, duplicate transaction IDs, duplicate payments, statistical outlier detection), keeps a persistent history of uploads, supports manual false-positive marking, and exports findings to CSV and PDF.

DevOps · Containerization & CI/CD

Dockerized Monitoring Tools with Automated CI/CD

Role Sole developer, end to end Stack Python, Docker, GitHub Actions, Docker Hub Repositories github.com/zeljkotr

Two monitoring tools moved from ad hoc scripts to a repeatable delivery process: containerized, version-controlled, and automatically built and published on every change.

The first is a server-availability monitor that pings a list of hosts on a schedule and emails an alert the moment one becomes unreachable, with deduplication so a flapping connection triggers a single notification rather than repeated alerts. The second is a Flask dashboard reporting live CPU, memory, disk, network, uptime, temperature, and active sessions for a server, accessible from any device on the network.

Both are built through a GitHub Actions pipeline that builds a Docker image on every push and publishes it to Docker Hub, ready to pull and run anywhere without a manual build step.

Repositories: Server-monitor-with-email-alert, system-monitor · Docker Hub: hub.docker.com/u/zeljkotr
